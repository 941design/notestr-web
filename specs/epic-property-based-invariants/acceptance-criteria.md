# Property-Based Invariant Tests ‚Äî Acceptance Criteria

These criteria are derived from `spec.md` and map to the stories in the spec's `## Stories` section. Each AC is observable from a test run (CI log or local invocation) and binds to one or more invariant IDs from the catalogue. ACs are grouped by story.

## Terminology

- **Invariant ID** ‚Äî `S1..S10`, `A1..A15`, `C0..C3`, `D1..D3`, `R1` as defined in `spec.md`. Tests reference these in comments at the top of each `it` / `test` block.
- **Property test** ‚Äî a `fc.assert(fc.property(...))` (sync) or `fc.assert(fc.asyncProperty(...))` (async) invocation with a `numRuns` setting.
- **Counterexample** ‚Äî fast-check's shrunk minimal failing input, printed as JSON-like output with a `seed` and `path` for reproduction.
- **Settled state** / **quiescence** ‚Äî defined in `spec.md` ¬ß Invariant Catalogue.

## Dependency Wiring (S1)

- **AC-DEP-1** ‚Äî `package.json` lists `fast-check` in `devDependencies` at version `^3.23.0` or newer compatible. `npm install` resolves with no peer-dep warnings.
- **AC-DEP-2** ‚Äî `Makefile` defines a `test-property` target that runs vitest with `FAST_CHECK_NUM_RUNS=10000` against the two property test files in `src/store/`.
- **AC-DEP-3** ‚Äî `make test` (existing target) is unchanged in semantics: it runs all `*.test.ts` files including the new `*.property.test.ts` ones, with default fast-check budget (1000 runs per property unless overridden).
- **AC-DEP-4** ‚Äî `make test-property` and `make test` both pass on a clean checkout immediately after S1 lands (no tests yet ‚Äî proves dependency wiring is correct).
- **AC-DEP-5** ‚Äî A failing property test (artificially injected, e.g. `expect(1).toBe(2)` inside `fc.property`) causes the make target to exit non-zero. Verified once, then reverted. (Sanity check on the build wiring; not committed.)

## Reducer Properties ‚Äî S1, S2, S3, S8, S9, A1‚ÄìA6, A13, R1 (S2 in spec)

- **AC-RED-1 (S1)** ‚Äî A property test asserts that for any sequence of up to 30 events generated by `arbEventSequence`, every `Task` in `replayEvents(events)` has `status ‚àà {open, in_progress, done, cancelled}`. Runs at least 1000 cases.
- **AC-RED-2 (S2)** ‚Äî A property test asserts every `Task` in any reachable state has non-empty `id`, `createdAt > 0`, `updatedAt >= createdAt`, and `assignee` either `null` or a string of length 64 of hex characters.
- **AC-RED-3 (S3)** ‚Äî A property test asserts that for any task `t` present in two intermediate states `s_i` and `s_j` (i < j) of a replay, `s_i.get(t.id).createdBy === s_j.get(t.id).createdBy` and `s_i.get(t.id).createdAt === s_j.get(t.id).createdAt`.
- **AC-RED-4 (S4)** ‚Äî A property test asserts that for every task in a final replayed state, the task's `updatedAt` equals the maximum `updatedAt` among all events in the input sequence whose `taskId` (or `task.id` for `task.created`) matches the task and which were not stale-rejected by the reducer.
- **AC-RED-5 (S8)** ‚Äî A property test using `createInMemoryKVStore` with two distinct group ids `g1` and `g2` asserts that after any interleaved sequence of `appendEvent(g1, ...)` and `appendEvent(g2, ...)` calls, `loadEvents(g1)` and `loadEvents(g2)` are disjoint with respect to event identity (object reference equality on the in-memory KV).
- **AC-RED-6 (S9)** ‚Äî A property test asserts that for any sequence of events appended to an in-memory KV under group `g`, `replayEvents(loadEvents(g))` deep-equals the in-memory state produced by folding `applyEvent` over the same sequence.
- **AC-RED-7 (A1)** ‚Äî A property test asserts that after `dispatch(actor, createTask(title, desc, actor.pubkey))` (modelled at the reducer layer as `applyEvent(state, { type: "task.created", task })`), the resulting state contains the new task with `status: "open"`, `assignee: null`, `createdBy === actor.pubkey`, `createdAt === updatedAt`.
- **AC-RED-8 (A2)** ‚Äî A property test asserts that for any state and any `task.status_changed` event with `event.updatedAt >= state[event.taskId].updatedAt`, the post-apply state has `state[event.taskId].status === event.status`.
- **AC-RED-9 (A3)** ‚Äî A property test asserts that for any state and any `task.updated` event with non-stale timestamp, every key `k` in `event.changes` satisfies `post[event.taskId][k] === event.changes[k]`, and every key `k'` not in `event.changes` (and not `updatedAt`) satisfies `post[event.taskId][k'] === pre[event.taskId][k']`.
- **AC-RED-10 (A4)** ‚Äî A property test asserts that for any state and any non-stale `task.assigned` event, `post[event.taskId].assignee === event.assignee` (including the explicit `null` case).
- **AC-RED-11 (A5)** ‚Äî A property test asserts that for any state and any non-stale `task.deleted` event, `event.taskId` is absent from `post`.
- **AC-RED-12 (A6)** ‚Äî A property test asserts that for any state and any of `task.updated`, `task.status_changed`, `task.assigned`, `task.deleted` event with `event.updatedAt < state[event.taskId].updatedAt`, the post-apply state for `event.taskId` is byte-identical to the pre-apply state. The other tasks may also be unchanged (and trivially so).
- **AC-RED-13 (A13)** ‚Äî A property test asserts that for any state `s` and any non-snapshot event `e`, `applyEvent(applyEvent(s, e), e)` deep-equals `applyEvent(s, e)`.
- **AC-RED-14 (R1)** ‚Äî A labelled property test asserts that for any state `s` containing a task `t`, after `applyEvent(s, { type: "task.deleted", taskId: t.id, updatedAt: t.updatedAt + 1, ... })` followed by `applyEvent(_, { type: "task.created", task: { ...t, updatedAt: t.updatedAt + 2 } })`, the final state contains a task with id `t.id`. The test labels this case via `fc.statistics` so any change in resurrection rate is visible in CI output.

## Reducer Convergence Subset ‚Äî C1, C2, C3 (single-client) (S3 in spec)

- **AC-RED-15 (C1, atomic-kind subset)** ‚Äî A property test asserts that for a list of `task.status_changed` events targeting a single task with strictly distinct timestamps, `replayEvents(events)` is invariant under permutation. Verified by comparing the post-replay state across two arbitrary orderings of the same multiset. The narrowing to `task.status_changed`-only mirrors spec.md ¬ßC1: the reducer's per-task-object `>=` guard makes mixed-kind sequences inherently order-dependent (see spec.md *Known LWW non-commutativity*). Other atomic-replace kinds (`task.assigned`, `task.deleted`) would be individually eligible by the same reasoning, but mixing kinds on a single task ‚Äî even atomic ones ‚Äî exposes the cross-kind divergence and is therefore excluded from this test.
- **AC-RED-16 (C2)** ‚Äî A property test asserts that re-labelling every event's `updatedBy` field (and `task.createdBy` for `task.created`) does not affect the set of task ids present in the final state, the assignee distribution, the status distribution, or any field other than `createdBy` / `updatedBy`. (`createdBy` is asserted unchanged from input via S3.)
- **AC-RED-17 (C3)** ‚Äî A property test asserts that for any event sequence, replaying with each event repeated 1‚Äì3 times yields a final state byte-identical to replaying once.

## Labelled Properties ‚Äî D1, D3 (S3 in spec)

- **AC-RED-18 (D1)** ‚Äî A property test asserts that for any state with a task `t`, given two `task.updated` events `e1` (by author U1) and `e2` (by author U2) both with `updatedAt === t.updatedAt + 1` and different `changes`, `replayEvents([t.creation, e1, e2])` and `replayEvents([t.creation, e2, e1])` may differ. The test does NOT assert equality (that would be wrong ‚Äî D1 is the divergent case). It records the divergence rate via `fc.statistics(arbInput, ({ states }) => states[0] !== states[1] ? "divergent" : "equal")`. CI run logs print this distribution.
- **AC-RED-19 (D3)** ‚Äî A property test asserts that for any state `s` reached by a sequence of individual events at timestamps `[t0, t0+1, ..., t0+n]`, applying a `task.snapshot` event whose `tasks` array reflects the state at `t0` afterward produces a state byte-identical to the snapshot ‚Äî i.e. all events after `t0` are lost. The test labels this as `"snapshot-overwrites-newer"` via `fc.statistics`.

## Multi-Client Convergence ‚Äî C0 (task subset), C1, C2, C3, A13, A15 (S4 in spec)

- **AC-MC-1 (C0, MLS-faithful delivery model)** ‚Äî A property test using `FakeBoard` with `numClients` ‚àà [2, 5] generates a single-writer event log (all dispatches from client 0) and a random per-client *causal prefix* ‚Äî i.e. each non-writer client receives a contiguous prefix `0..maxIdx` of the bus before `quiesce()` drains the remainder in shared total order. The schedule asserts that `clients[i]` deep-equals `clients[0]` for every `i` post-quiesce. Runs at least 500 cases. The model ‚Äî same total order to every client, different pause points ‚Äî is faithful to the real notestr system, where MLS provides a single total order per group; arbitrary per-client reorder is documented as architecturally out of scope per spec.md ¬ßC0 and *Known LWW non-commutativity*.
- **AC-MC-2 (C1, atomic-kind subset)** ‚Äî A property test generates a single task plus a fixed multiset of `task.status_changed` events for that task with strictly distinct timestamps, dispatches them from client 0, and for each non-writer client delivers the status events in a randomly permuted order. Asserts post-delivery equality across clients regardless of permutation. The narrowing to `task.status_changed`-only is the same restriction as AC-RED-15 and for the same reason ‚Äî mixed-kind sequences expose the per-task-object LWW guard's order-dependence.
- **AC-MC-3 (C2)** ‚Äî A property test re-labels every event's `updatedBy` to a different uniformly-random pubkey before delivery, asserts the resulting clients converge to a state whose tasks differ from the un-relabelled run only in `createdBy` / `updatedBy` fields.
- **AC-MC-4 (C3)** ‚Äî A property test in which each `bus` slot is delivered to each client between 1 and 3 times (uniformly chosen) asserts the post-quiescence state matches the once-each-deliver baseline.
- **AC-MC-5 (A13)** ‚Äî A property test asserts that delivering the same event to the same client multiple times produces no observable change after the first delivery.
- **AC-MC-6 (A15)** ‚Äî A property test with `numClients = 2`, both representing the same identity (same `createdBy` / `updatedBy` pubkey on dispatched events), asserts post-quiescence equality between the two clients. This is the multi-device same-pubkey case.

## Labelled Properties Multi-Client ‚Äî D1, D3 (S5 in spec)

- **AC-MC-7 (D1)** ‚Äî A property test introduces equal-timestamp updates from two clients to the same task and records the inter-client divergence rate via `fc.statistics`. Visible in CI logs; does NOT fail.
- **AC-MC-8 (D3)** ‚Äî A property test forces a `task.snapshot` to be delivered after later individual events on at least one client and records the resulting per-client divergence via `fc.statistics`.

## Full-Stack Property Spec ‚Äî S5, S6, S7, S10, A7, A8, A9, A10, A11, A12, A14, C0 (S6 in spec)

- **AC-FS-1 (file existence)** ‚Äî `e2e/tests/multi-user.property.spec.ts` exists and is included by Playwright's default test discovery (matches `playwright.config.ts` `testDir`).
- **AC-FS-2 (test discovery)** ‚Äî `make e2e` runs the new file as part of the suite. `npx playwright test multi-user.property.spec.ts` runs the suite alone.
- **AC-FS-3 (commands cover the DSL)** ‚Äî The file defines one `fc.AsyncCommand` class per DSL verb in scope: `CgCommand`, `InCommand`, `LgCommand`, `FdCommand`, `RdCommand`, `CtCommand`, `UtCommand`, `ScCommand`, `AsCommand`, `UnCommand`, `DtCommand`, `RlCommand`, `SwCommand`. (`Au` and `Dc` are setup, not random commands.)
- **AC-FS-4 (numRuns)** ‚Äî The single property under `test.describe.serial` runs with `numRuns: 20` and `maxCommands: 10` per chain. End-to-end wall-clock under `make e2e` for the property test alone is at most 10 minutes on the CI host.
- **AC-FS-5 (C0 settled-state equality)** ‚Äî After every `fc.commands` run, the test calls `await real.quiesce()` and asserts that for every group in the model, every current member's view (read via `__notestrTestTasks`, `__notestrTestGroups`, `getGroupMembers`, `getPubkeyLeafNodes`) is identical to every other current member's view, in all four dimensions (tasks, members, epoch, leaf count).
- **AC-FS-6 (S5 member-iff-leaf)** ‚Äî After every `fc.commands` run, for every group `g` and pubkey `p` observed in any context, the test asserts `(p ‚àà getGroupMembers(g.state)) === (getPubkeyLeafNodes(g.state, p).length >= 1)`.
- **AC-FS-7 (S6 epoch monotonicity)** ‚Äî Each command's `run` records the post-action `epoch` per context. The test asserts the recorded epoch sequence per context is non-decreasing.
- **AC-FS-8 (S7 identity isolation)** ‚Äî Any `SwCommand` postcondition asserts that the post-switch context's `__notestrTestTasks` returns zero tasks belonging to groups the previous identity was a member of.
- **AC-FS-9 (S10 device-list / leaf-count consistency)** ‚Äî At the end of every `fc.commands` run, for every pubkey `p` rendered in `DeviceList.tsx`, the count of `data-testid="device-row"` rows equals `getPubkeyLeafNodes(g.state, p).length`. Verified via `page.locator(...).count()`.
- **AC-FS-10 (A7..A12 per-command postconditions)** ‚Äî Each `Command` class's `run` method includes an `expect(...)` assertion verifying its action's postcondition (A1..A12 as relevant) before returning. Failure of any postcondition shrinks via fast-check.
- **AC-FS-11 (A14 negative delivery)** ‚Äî `LgCommand` and `FdCommand` (last-leaf case) postconditions include a 2-second observer wait via `e2e/fixtures/ndk-subscriber.ts` confirming no new kind-445 events for the affected group are decryptable on the leaving / forgotten context.
- **AC-FS-12 (counterexample readability)** ‚Äî When fast-check shrinks a counterexample, the failing chain prints to the test output as a sequence of `Actor.Verb(args)` strings matching the matrix DSL. Each `Command` class implements `toString()` returning the DSL form. Verified manually by injecting a deliberate failure.
- **AC-FS-13 (`describe.serial` discipline)** ‚Äî The property test sits inside a single `test.describe.serial` block. Sibling describes (if any) do not depend on its outcome. Per the project rule, this is correct because `fc.commands` is genuinely a chain ‚Äî postcondition validity depends on all prior commands.

## Cross-Cutting Invariants

- **AC-X-NAMING-1** ‚Äî Every property test's body begins with a single comment line of the form `// {ID}: {short description}` referencing the invariant ID(s) being asserted. Multi-invariant tests list all IDs comma-separated.
- **AC-X-NAMING-2** ‚Äî Every `it` / `test` title contains the invariant ID(s) being asserted in brackets at the start, e.g. `it("[S1] status enum closure")`. Greppable by ID.
- **AC-X-RUNS-1** ‚Äî Layer-1 (reducer) property tests use `numRuns: 1000` by default. CI logs include the configured number per test (fast-check prints it on success when `verbose: 1`).
- **AC-X-RUNS-2** ‚Äî Layer-2 (multi-client) property tests use `numRuns: 500` by default.
- **AC-X-RUNS-3** ‚Äî Layer-3 (full-stack) property tests use `numRuns: 20` by default.
- **AC-X-DET-1** ‚Äî When a property fails, the printed counterexample includes a `seed` value sufficient to deterministically reproduce the failure. Re-running with `FAST_CHECK_SEED=<seed>` reproduces the same counterexample.
- **AC-X-NO-PROD-CHANGE-1** ‚Äî No file under `src/marmot/`, `src/store/task-events.ts`, `src/store/task-reducer.ts`, `src/store/task-store.tsx`, `src/store/persistence.ts`, or `src/components/` is modified by this epic. (The dispatch path, reducer, and persistence are tested as-is.) Verified by `git diff --stat origin/master...` showing changes only under `src/store/*.property.test.ts`, `src/marmot/*` is untouched, `e2e/tests/multi-user.property.spec.ts`, `docs/`, `specs/`, `package.json`, `Makefile`.
- **AC-X-NO-EXAMPLE-DELETION-1** ‚Äî No existing example-based test (`*.test.ts` or `*.spec.ts`) is deleted, renamed, or `.skip`-ed by this epic. Property tests are additive.
- **AC-X-CI-1** ‚Äî Layer-1 and Layer-2 property tests run on every CI build via `make test`. Layer-3 runs as part of `make e2e` (existing CI gate).
- **AC-X-CI-2** ‚Äî A failing property test fails the CI build with a non-zero exit and a printed counterexample. A labelled-only property (D1/D3) printing high divergence does NOT fail the build.
- **AC-X-OBSERVABILITY-1** ‚Äî Every Layer-1 and Layer-2 file uses `fc.statistics` to report at least one distribution per major property (e.g. action-type frequency, divergence rate for D properties, average action-chain length). Visible in CI logs when run with `--reporter=verbose`.
